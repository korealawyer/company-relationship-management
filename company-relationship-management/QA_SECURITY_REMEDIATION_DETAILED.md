# 🛡️ CRM QA 및 보안 사각지대 최종 개선 상세 내역 (총 15건)

**문서 개요**:  
본 문서는 IBS 로펌 기업관계관리 CRM(Next.js 16)의 도입을 앞두고 진행된 여러 차례의 **QA(교차 검증) 및 API 보안 강화 세션**에서 도출된 모든 취약점과 테스트 사각지대(총 15건)에 대한 상세 조치 내역입니다.

---

## 🏗️ 1차 개선: 치명적 취약점 및 보안 인프라 패치 (총 9종)

1. **내부 통신 전용 JWT 발급 검증 (`/api/auth/jwt`)**: 
   - `x-internal-secret` 헤더를 검증하여 외부에서의 악의적인 권한 탈취(Role Spoofing) 시도를 차단했습니다.
2. **Origin 미전송 요청 차단**: 
   - JWT 서명 요청 시 클라이언트의 Origin 헤더가 존재하지 않는 비정상/스크립트 기반 요청을 403 에러로 즉시 튕겨내도록 패치했습니다.
3. **빈 시크릿 서명 취약점 원천 방어**: 
   - 서버 환경변수 리셋 등으로 인해 빈 문자열(`JWT_SECRET=""`)로 JWT가 하드코딩 서명되는 보안 사고를 원천 차단했습니다.
4. **엔드포인트별 Role 통제 강제 하드코딩 (`/api/drip`, `/api/notion`)**: 
   - 핵심 API들에 미들웨어를 거치지 않고 직접 세션 유효성과 해당 직무(Sales, Admin, Lawyer) 권한을 재확인하는 로직을 고정 삽입했습니다.
5. **IDOR 취약점 패치 (결제조회) (`/api/payment/check`)**: 
   - 결제 내역 조회 시, 로그인한 유저의 본인 회사(`companyId`) 데이터만 조회되도록 백엔드 교차 검증 로직을 추가했습니다.
6. **IDOR 취약점 패치 (리드갱신) (`PATCH /api/leads`)**: 
   - 타인의 담당 리드 데이터 식별자를 임의로 넣어 데이터를 덮어씌울 수 없도록 담당자 소유권 교차 검증을 추가했습니다.
7. **XSS(크로스 사이트 스크립팅) 인젝션 방어 (`/api/email`, `/api/drip`)**: 
   - 변호사노트 및 이메일 템플릿 생성 시, 악성 HTML 스크립트가 파싱되지 못하도록 `sanitizeHtml` 강제 이스케이프 처리를 적용했습니다.
8. **서버 런타임 크래시 방어 (Malformed JSON)**: 
   - 모든 API의 `req.json()` 파싱부에 `try-catch` 블록을 씌워 고의적인 불량 JSON 페이로드로 인해 서버가 꺼지는 500 디버그 에러 현상을 막고 정상적인 400 에러로 방어합니다.
9. **Rate Limiting 고도화 도입**: 
   - Upstash 및 In-memory 기반 Rate Limiter를 도입하여 분당 제한을 초과하면 429 에러를 반환하며, 식별 Key를 단순 IP가 아닌 `${api_name}_${ip}_${auth.role}` 조합으로 세분화해 무차별 대입을 완벽히 방어했습니다.

## 🚀 2차 개선: 백로그 및 엣지 케이스 완벽 구현 (총 6종)

10. **초대코드 Timezone 만료 보정 버그 픽스 (`auth.ts`)**: 
    - 서버 타임존 파싱 차이로 인해 자정에 초대코드가 하루 일찍 닫혀버리는 현상을 `.setHours(23, 59, 59, 999)`로 강제 설정하여 하루 유효기간을 꽉 채우도록 보정했습니다.
11. **SSE 스트리밍 제공자 분기 로직 정상화 (`chat-stream`)**: 
    - Anthropic API 키가 없으면 무조건 Mock 데이터로 빠지던 챗봇 로직을 변경하여, `AI_PROVIDER=openai` 설정 시 실제 OpenAI 스트리밍이 병행 동작하도록 라우팅을 고도화했습니다.
12. **CSV 대용량 업로드 메모리 압사 방지 (`leads/upload`)**: 
    - Vercel Serverless Edge 환경의 메모리 고갈(OOM)을 막기 위해 CSV 파일 업로드의 Payload 용량을 최대 5MB로 제한 및 검증하는 가드를 세웠습니다.
13. **AI 사용 로그 메모리 누수 픽스 (`aiUsage.ts`)**: 
    - 런타임 서버 메모리에 누적되는 AI 사용 로그가 끝없이 쌓이지 않도록 `MAX_LOG_SIZE`를 500건으로 잡고, 500건 초과 시 선입선출(`shift()`) 방식으로 가장 오래된 데이터를 버리는 구조를 확인 및 구현했습니다.
14. **다중 탭(Storage) 로그아웃 무결성 동기화 (`AuthContext.tsx`)**: 
    - 사용자가 브라우저 탭을 여러 개 연 상태에서 다른 탭에서 계정을 전환하거나 로그아웃했을 때, 서로 꼬임 없이 보고 있는 현재 탭도 즉시 권한을 잃고 동기화되도록 이벤트 리스너 조치를 완료했습니다.
15. **회귀 검증(Playwright E2E) 테스트 스크립트화 (`security.spec.ts`)**: 
    - 패치된 취약점(Rate Limit 429 정상 동작, IDOR 403 튕김 지점, 불량 JSON 에러 제어 등)들이 향후 코드가 바뀌어도 정상 방어되는지 CI/CD 단계에서 상시 검사해 주는 E2E 멀티 자동화 코드를 짰습니다.

## 🔐 3차 개선: QA 교차 검증 사각지대 및 결제/인증 아키텍처 보완 (총 6종)

16. **타입 기반 Injection 500 에러 크래시 방어 (`/api/auth/login`)**: 
    - `req.json()` 파싱 이후 단순 구조 분해 할당의 허점(`TypeError`)을 노린 악의적 페이로드를 차단하기 위해 엄격한 데이터 타입(Type) 유효성 검증 레이어를 추가했습니다.
17. **비밀번호 무차별 대입(Brute-force) 타겟 방어 (`/api/auth/login`)**: 
    - 단순 IP 기반 요청 제한을 넘어, 단일 계정(이메일/사업자번호)에 대한 지속적 공격을 막기 위해 **Account 단위 Rate Limit**을 이중 구축하여 암호 크래킹을 방어했습니다.
18. **하드코딩 Fallback 서명 취약점 원천 제어 (`/api/auth/login`)**: 
    - 프로덕션 환경에서 `JWT_SECRET` 환경 변수 누락 시 취약한 디폴트 시크릿으로 토큰이 발급되어 버리던 심각한 보안 결함을 찾아내 원천 차단(Fail-safe) 처리했습니다.
19. **화이트리스트 기반 IDOR 완벽 통제 (`/api/payment/check`)**: 
    - 조회 권한이 없는 내부 직원이나 일반 권한 계정이 임의로 타사 결제 데이터를 열람할 수 없도록, **수퍼 관리자, 영업팀, 본인 소속 고객사**만 조회할 수 있는 안정한 화이트리스트 방식으로 교체했습니다.
20. **JWT Blacklist (강제 로그아웃/제명) 아키텍처 도입 (`src/lib/tokenBlacklist.ts`)**: 
    - 무상태(Stateless) 쿠키의 단점인 서버 통제 불가를 해결하기 위해 토큰 식별자(`jti`) 기반 블랙리스트 아키텍처를 도입하여, 권한이 정지되거나 탈취된 토큰을 즉각 401 차단하는 방어망을 구현했습니다.
21. **PG Webhook 서명 위변조 방어 및 멱등성 보장 (`/api/payment/webhook`)**: 
    - 향후 PG사 연동 시 가짜 결제 승인을 막기 위한 **HMAC SHA-256 서명 위변조 검증(`x-pg-signature`)** 및 다중 중복 결제 요청을 안전하게 무시하는 **트랜잭션 멱등성(Idempotency) 구조**를 교차 검증하여 스캐폴딩했습니다.
