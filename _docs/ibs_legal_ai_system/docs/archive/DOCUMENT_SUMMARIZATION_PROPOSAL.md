# 문서 요약 단계 제안서 (Document Summarization Proposal)

## 📋 현재 상황 분석

### 현재 워크플로우
1. **문서 검색** (`_search_documents_node`):
   - RAG 워크플로우를 통해 관련 문서 검색 (기본 5개, 최대 20개 이상)
   - 검색된 문서의 **전체 텍스트**가 `context` 문자열로 구성됨
   - 각 문서의 chunk 텍스트가 그대로 포함됨

2. **프롬프트 생성** (`_generate_prompt_node`):
   - `context` (전체 문서 텍스트)가 프롬프트에 직접 포함됨
   - `_build_structured_blog_prompt`에서 `{CONTEXT}` 플레이스홀더에 전체 문서 내용 삽입

3. **콘텐츠 생성** (`_generate_draft_node`):
   - 전체 문서 텍스트가 포함된 프롬프트를 GPT에 전달
   - **문제점**: 프롬프트가 매우 길어짐 (수천~수만 토큰)

### 현재 문제점
- **토큰 사용량 과다**: 검색된 문서의 전체 텍스트가 프롬프트에 포함되어 토큰 사용량이 많음
- **핵심 정보 희석**: 전체 문서 중 실제로 필요한 핵심 정보만 추출하는 것이 어려움
- **비용 증가**: 긴 프롬프트로 인한 API 비용 증가
- **응답 품질 저하 가능성**: 너무 많은 정보로 인해 LLM이 핵심에 집중하기 어려울 수 있음

## 💡 제안: LLM 기반 문서 요약 단계 도입

### 핵심 아이디어
**문서 검색 후, 콘텐츠 생성 전에 검색된 문서들을 GPT를 통해 핵심만 왜곡 없이 요약하는 단계를 추가**

### 제안하는 워크플로우 구조

```
[기존]
문서 검색 → 프롬프트 생성 → 콘텐츠 생성

[제안]
문서 검색 → **문서 요약** → 프롬프트 생성 → 콘텐츠 생성
```

### 상세 워크플로우

1. **문서 검색** (`_search_documents_node`):
   - 기존과 동일하게 RAG 검색 수행
   - 검색된 문서의 전체 텍스트를 `state["context"]`에 저장

2. **문서 요약** (`_summarize_documents_node`) **[신규]**:
   - 검색된 문서들을 GPT에 전달하여 핵심 정보만 요약
   - 요약 프롬프트에 **왜곡 방지 원칙** 명시
   - 요약 결과를 `state["summarized_context"]`에 저장
   - 원본 문서 메타데이터(법령 조문 번호, 판례 번호 등)는 보존

3. **프롬프트 생성** (`_generate_prompt_node`):
   - `context` 대신 `summarized_context` 사용
   - 원본 `references` 정보(법령/판례 번호)는 그대로 유지

4. **콘텐츠 생성** (`_generate_draft_node`):
   - 요약된 컨텍스트를 기반으로 콘텐츠 생성

## ✅ 예상 효과

### 1. 토큰 사용량 감소
- **현재**: 검색된 문서 전체 텍스트 (예: 5개 문서 × 평균 2000자 = 10,000자 이상)
- **요약 후**: 핵심 정보만 추출 (예: 2000~3000자)
- **절감률**: 약 70~80% 토큰 감소 예상

### 2. 비용 절감
- 프롬프트 토큰 감소로 API 비용 절감
- 요약 단계 추가 비용 < 전체 프롬프트 토큰 감소로 인한 절감액

### 3. 콘텐츠 품질 향상
- 핵심 정보에 집중하여 더 정확하고 일관된 콘텐츠 생성 가능
- 불필요한 정보 제거로 LLM의 혼란 감소

### 4. 응답 속도 개선
- 짧은 프롬프트로 인한 응답 시간 단축 가능성

## ⚠️ 고려사항 및 리스크

### 1. 정보 손실 위험
- **리스크**: 요약 과정에서 중요한 세부 정보가 누락될 수 있음
- **완화 방안**:
  - 요약 프롬프트에 "핵심 법령 조문 번호, 판례 번호, 법정형, 구성요건은 반드시 포함" 명시
  - 요약 결과에 원본 문서 메타데이터(법령/판례 번호) 보존
  - 요약 길이 제한을 적절히 설정 (너무 짧지 않게)

### 2. 왜곡 위험
- **리스크**: 요약 과정에서 법률 내용이 왜곡될 수 있음
- **왜화 방안**:
  - 요약 프롬프트에 "왜곡 없이 정확히 요약" 원칙 명시
  - 요약 프롬프트에 "참고 문서의 내용만 요약, 추가 해석 금지" 명시
  - 요약 결과 검증 로직 추가 (선택사항)

### 3. 추가 비용 및 지연
- **리스크**: 요약 단계 추가로 인한 API 호출 1회 증가 및 지연 시간 증가
- **완화 방안**:
  - 요약은 `gpt-4-turbo-preview`보다 빠른 모델 사용 고려 (예: `gpt-3.5-turbo`)
  - 요약 결과 캐싱 고려 (같은 문서 조합에 대해)
  - 전체 프롬프트 토큰 감소로 인한 절감액이 요약 비용보다 클 것으로 예상

### 4. 법령 조문/판례 번호 보존
- **리스크**: 요약 과정에서 법령 조문 번호나 판례 번호가 누락될 수 있음
- **완화 방안**:
  - 요약 프롬프트에 "법령 조문 번호(예: 형법 제347조), 판례 번호(예: 대법원 2019도12345)는 반드시 포함" 명시
  - 원본 `references` 정보는 별도로 보존하여 프롬프트에 포함

## 🎯 구현 방안 (High-Level)

### 1. 새로운 노드 추가: `_summarize_documents_node`

**입력**:
- `state["context"]`: 검색된 문서의 전체 텍스트
- `state["references"]`: 문서 메타데이터 (법령/판례 번호 등)
- `state["topic"]`: 주제

**처리**:
- 요약 전용 프롬프트 생성:
  ```
  아래 검색된 법률 문서들을 주제 "{topic}"에 맞게 핵심만 요약해주세요.
  
  요약 원칙:
  1. 왜곡 없이 정확히 요약
  2. 법령 조문 번호, 판례 번호는 반드시 포함
  3. 법정형, 구성요건 등 핵심 법률 정보는 누락하지 않음
  4. 참고 문서의 내용만 요약, 추가 해석 금지
  5. 각 문서의 핵심 내용을 간결하게 정리
  
  검색된 문서:
  {context}
  ```
- GPT 호출하여 요약 수행
- 요약 결과를 `state["summarized_context"]`에 저장

**출력**:
- `state["summarized_context"]`: 요약된 문서 내용

### 2. LangGraph 워크플로우 수정

**기존 그래프**:
```
search_documents → generate_prompt → generate_draft → ...
```

**수정된 그래프**:
```
search_documents → summarize_documents → generate_prompt → generate_draft → ...
```

### 3. 프롬프트 생성 로직 수정

**기존**:
```python
context = state.get("context", "")
prompt = self._build_structured_blog_prompt(
    topic=topic,
    context=context,  # 전체 문서 텍스트
    ...
)
```

**수정 후**:
```python
# 요약된 컨텍스트 우선 사용, 없으면 원본 사용 (fallback)
summarized_context = state.get("summarized_context")
context = summarized_context if summarized_context else state.get("context", "")

prompt = self._build_structured_blog_prompt(
    topic=topic,
    context=context,  # 요약된 문서 텍스트
    ...
)
```

### 4. 요약 프롬프트 템플릿 파일 생성

**새 파일**: `src/rag/prompts/summarization_prompt.txt`
- 요약 원칙
- 왜곡 방지 지침
- 필수 포함 사항 (법령/판례 번호 등)

## 🔄 대안 방안

### 대안 1: 선택적 요약 (Threshold 기반)
- 컨텍스트 길이가 특정 임계값(예: 5000자)을 초과할 때만 요약 수행
- 짧은 컨텍스트는 요약 생략하여 비용 절감

### 대안 2: 문서별 개별 요약
- 각 문서를 개별적으로 요약한 후 병합
- 문서 간 관계 파악이 어려울 수 있음

### 대안 3: 하이브리드 접근
- 법령 조문: 원문 유지 (짧고 정확해야 함)
- 판례/실무 문서: 요약 수행
- 타입별로 다른 전략 적용

### 대안 4: 요약 결과 캐싱
- 같은 문서 조합에 대해 요약 결과 캐싱
- 캐시 히트 시 요약 단계 생략

## 📊 예상 성능 지표

### 토큰 사용량 비교 (예시)

**현재 (5개 문서, 평균 2000자/문서)**:
- 입력 토큰: ~10,000 토큰 (문서 전체)
- 출력 토큰: ~2,000 토큰
- **총 토큰**: ~12,000 토큰

**요약 적용 후**:
- 요약 단계:
  - 입력 토큰: ~10,000 토큰 (문서 전체)
  - 출력 토큰: ~1,000 토큰 (요약)
  - **요약 단계 총 토큰**: ~11,000 토큰
- 콘텐츠 생성 단계:
  - 입력 토큰: ~3,000 토큰 (요약본 + 프롬프트)
  - 출력 토큰: ~2,000 토큰
  - **생성 단계 총 토큰**: ~5,000 토큰
- **총 토큰**: ~16,000 토큰

**초기 분석**: 요약 단계 추가로 인해 총 토큰이 증가할 수 있음. 하지만:
- 요약은 더 빠른/저렴한 모델 사용 가능 (`gpt-3.5-turbo`)
- 요약 결과 캐싱으로 반복 요청 시 비용 절감
- 콘텐츠 품질 향상으로 재작성 횟수 감소 가능

### 최적화 후 예상 (요약 모델: gpt-3.5-turbo, 캐싱 적용)

**첫 요청**:
- 요약 단계: ~11,000 토큰 (gpt-3.5-turbo, 저렴)
- 생성 단계: ~5,000 토큰 (gpt-4-turbo-preview)
- **총 비용**: 기존 대비 비슷하거나 약간 증가

**캐시 히트 시**:
- 요약 단계: 0 토큰 (캐시 사용)
- 생성 단계: ~5,000 토큰
- **총 비용**: 기존 대비 약 60% 절감

### 최종 권장사항

✅ **도입 권장**: 문서 요약 단계 도입을 권장합니다.

**이유**:
1. **콘텐츠 품질 향상**: 핵심 정보에 집중하여 더 정확한 콘텐츠 생성 가능

4. **유지보수성**: 요약 로직을 별도 노드로 분리하여 관리 용이

**주의사항**:
- 요약 프롬프트에 **왜곡 방지 원칙**을 명확히 명시
- 법령/판례 번호 등 **핵심 메타데이터 보존** 필수
