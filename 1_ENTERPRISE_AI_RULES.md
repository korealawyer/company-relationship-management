# 🚀 [절대 원칙] 엔터프라이즈 SaaS 개발 AI 가이드라인 (Master Rules)

이 문서는 본 프로젝트에서 AI 어시스턴트(안티그래비티 등)가 코드를 작성, 수정, 기획할 때 반드시 지켜야 하는 **최상위 헌법**입니다. 어떠한 경우에도 이 원칙을 무시하고 빠르고 더러운(Quick and Dirty) 코드를 작성해서는 안 됩니다.

---

## 🏗️ 1. 아키텍처 규칙: 기능 분할 설계 (Feature-Sliced Design)
프로젝트가 방대해짐에 따라 스파게티 코드를 방지하기 위해 엄격한 폴더 구조를 유지합니다.

- **규칙 1 (라우트 분리):** `app/` 폴더 안에는 오직 라우팅 껍데기(`page.tsx`, `layout.tsx`)만 존재해야 합니다. 여기서 방대한 UI를 직접 그리지 마세요. 비즈니스 로직은 철저히 배제합니다.
- **규칙 2 (기능 모듈화):** 모든 핵심 비즈니스 로직과 UI는 `src/features/` 하위에 도메인/기능별로 분리하여 작성합니다. (예: `features/billing`, `features/sales`)
- **규칙 3 (공통 컴포넌트):** 여러 기능에서 재사용되는 버튼, 모달, 유틸리티 등은 `src/shared/` 하위에 작성합니다.

---

## 🔄 2. 서버 주도형 UI (Admin-Driven UI / End-to-End Sync)
변호사 포탈과 고객 포탈 간의 완벽한 기능 연동을 위해 클라이언트 하드코딩을 금지합니다.

- **규칙 1 (기능의 데이터화):** 프론트엔드 A(관리자)와 프론트엔드 B(고객)에 각각 별도로 기능을 하드코딩하여 끼워넣지 마세요.
- **규칙 2 (틀 렌더링):** 고객 포탈 등의 엔드유저 화면은 **오직 Supabase DB(예: `portal_features` 테이블)의 설정값(is_active 등)을 읽어와서 동적으로 렌더링하는 틀(Container) 역할**만 수행해야 합니다.
- **규칙 3 (관리자 제어):** 새로운 기능이 추가될 때, 관리자(변호사) 포탈에는 DB의 상태를 토글/변경할 수 있는 제어 스위치 로직만 구현합니다.

---

## 🗄️ 3. 데이터 및 API 원칙 (DB-First & Type Safety)
하이드레이션 이슈와 런타임 에러를 원천 차단하기 위한 무결성 원칙입니다. (ZERO_MOCKUP_GUIDE 확장)

- **규칙 1 (가짜 데이터 절대 금지):** 로컬 컴포넌트 내에 `const DUMMY_DATA = []` 형태로 배열이나 객체 목업을 하드코딩하는 짓을 100% 금지합니다.
- **규칙 2 (DB First):** 빈 화면부터 만들지 마세요. 기능을 구현하기 전에 반드시 Supabase 마이그레이션과 `seed.sql`을 통해 로컬 DB 환경부터 세팅하고 시작합니다.
- **규칙 3 (타입 제너레이션):** 수동으로 TypeScript `interface`를 작성하지 마세요. 반드시 Supabase CLI를 통해 DB에서 `gen types typescript` 명령어로 자동 추출된 타입을 최우선으로(`shared/types/database.types.ts`) 사용합니다.
- **규칙 4 (Server Action & Zod):** 클라이언트에서 직접 DB를 변형하는 것을 지양하고, Next.js `Server Actions`를 거치며, 파라미터는 무조건 `Zod` 스키마로 검증합니다.

---

## ⚡ 4. 상태 관리 규칙 (Server vs Client State)
단순하고 명확한 데이터 흐름을 강제합니다.

- **데이터 조회 (Read):** 최대한 `React Server Components (RSC)`에서 직결로 DB 데이터를 Fetch하여 내려줍니다.
- **데이터 변이 및 캐싱 (Mutation):** 클라이언트에서 상태를 변경하거나 데이터를 다시 불러올 때는 복잡한 `useEffect`를 쓰지 않고, 반드시 `TanStack Query (React Query)`를 사용하여 캐싱과 로딩 상태를 위임합니다.
- **전역 UI 상태:** 모달 열림여부, 다크모드 등 순수 UI 전역 상태에만 `Zustand`를 제한적으로 사용합니다.

---

**🤖 AI 행동 지침:**
이 문서를 인지한 순간부터, AI는 대표님의 지시("결제 내역 페이지 만들어줘")를 받을 때 위 원칙을 역산하여 **"DB 테이블 설계 -> 타입 생성 -> Server Action 로직 구현 -> (필요시 Admin 토글 구현) -> 클라이언트 동적 UI 컴포넌트(features 폴더) 조립 -> app 폴더에 연결"** 순서로 스스로 작업 계획을 세우고 실행해야 합니다.
