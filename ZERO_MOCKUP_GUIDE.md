# 🚨 [중요 규칙] Zero-Mockup & DB-First 개발 가이드라인

## 1. 배경 및 금지 사항 (우리는 왜 목업 데이터를 쓰지 않는가?)
프로젝트 배포 시 로컬에서 작성한 하드코딩된 가상 데이터(목업)와 실제 DB 데이터 연동 코드가 Vercel 환경에서 혼재되어, 사용자 화면에 가짜 데이터와 진짜 데이터가 번갈아 나타나는 치명적인 하이드레이션(Hydration) 꼬임 문제가 발생한 이력이 있습니다. 
따라서 본 프로젝트에서는 **코드 베이스 내에 하드코딩된 가짜 데이터(배열, JSON 등)를 심어두는 행위를 100% 금지**합니다.

## 2. 왜 이런 문제가 배포 환경에서 발생하는가? (기술적 원인)
1. **Fallback 로직의 오작동**: DB 응답이 0.1초라도 지연되면 예외 처리된 목업 데이터가 즉시 화면에 렌더링 됩니다.
2. **SSG/ISR 빌드 타임의 한계**: Vercel 빌드 시점에 DB 연결이 불안정하면 목업 데이터를 읽어 HTML로 고정시켜 버립니다.
3. **웹팩(Webpack) 번들링**: 컴포넌트 상단에 정의된 `const dummyData = []`는 배포용 자바스크립트 파일에 물리적으로 포함되어 영구적인 용량 낭비와 혼선을 야기합니다.
4. **AI의 '에러 회피성' 코딩**: 화면을 깨뜨리지 않기 위해 AI가 임의로 집어넣은 로컬 목업 코드는 추후 인간이 모두 찾아내어 지우기 매우 어렵습니다(독에 걸린 현상).

---

## 3. 🚀 새로운 개발 방법론 (이렇게 개발하세요)

### Step 1. 개발 전 DB 설계 및 세팅 최우선 (DB-First)
- 프론트엔드 UI를 먼저 그리기 전에 **반드시 Supabase(또는 실제 DB)의 테이블과 스키마부터 완전히 구성**합니다.

### Step 2. 로컬 테스트용 더미 데이터는 DB에 직투입
- 화면 UI 구성을 위해 더미 데이터가 필요하다면, 절대 로컬 코드 파일에 만들지 마세요.
- **Supabase DB의 실제 테이블에 테스트용 인서트를 실행하여 데이터를 삽입**해 둡니다.

### Step 3. 프론트엔드는 무조건 DB만 바라본다 (Single Source of Truth)
- 로컬 환경(localhost:3000)에서 개발할 때도, Vercel 배포 환경과 동일하게 곧바로 **Supabase API를 호출(Fetch)하여 데이터를 불러오도록(Pull) 코드를 작성**합니다.
- 데이터 로딩 중에는 목업을 보여주는 것이 아니라, 스켈레톤 UI(Loading Spinner)를 사용합니다.

## 🤖 [AI 코딩 어시스턴트(당신)를 위한 행동 지침]
이 프롬프트를 읽은 AI 어시스턴트는 앞으로의 모든 코드 작성 및 수정 시 아래 원칙을 강제로 지킵니다.

1. **하드코딩 절대 금지**: 더미 데이터 객체나 배열을 생성해서 컴포넌트에 주입하지 마세요.
2. **무조건 DB Fetch 로직 작성**: 데이터가 필요한 UI를 만들 때는 항상 Supabase 통신 함수를 기반으로 코드를 작성하세요.
3. **오류 상황 대처**: 데이터가 비어있다면 가짜 데이터를 넣어서 해결하지 말고, `데이터가 없습니다`라는 빈 상태(Empty State) 디자인을 렌더링하거나 로딩 상태로 처리하세요.
